# План виправлення помилок збірки

## Що саме зламано
GitHub Actions падає не через Tauri-конфіг чи workflow, а через Rust dependency resolution у `desktop/src-tauri`.

Лог прямо показує проблему:
- `brotli-8.0.3` компілюється з очікуванням `alloc-no-stdlib-2.0.4`
- одночасно в графі залежностей присутній `alloc-no-stdlib-3.0.0`
- через це `StandardAlloc` реалізує `Allocator<T>` з однієї major-версії, а `brotli` очікує trait з іншої
- результат: серія `error[E0277] ... multiple different versions of crate alloc_no_stdlib in the dependency graph`

Це видно в логу з файлу `pasted-2026-06-14T18-02-12-276Z.txt`:
- `brotli-8.0.3` падає в `combined_alloc.rs`
- одночасно згадуються `alloc-no-stdlib-2.0.4` і `alloc-no-stdlib-3.0.0`

## Чому попередні фікси були нестабільні
Просте перемикання між:
- `alloc-no-stdlib = "2"`
- `alloc-no-stdlib = "3"`

не є надійним методом, бо воно лікує лише симптом і може:
- залишати конфлікт у lockfile/резолві
- примушувати Cargo підбирати неприродну комбінацію залежностей
- поводитись по-різному локально й у GitHub Actions через кеш/оновлення індексу

Зовнішні джерела по crates підтверджують, що актуальні `brotli` і `brotli-decompressor` мають узгоджуватись на `alloc-no-stdlib ^2.0`, тож поява `3.0.0` у графі — це наслідок резолву/оновлення іншою залежністю або зафіксованого стану, а не те, що `brotli` треба переводити на `3`.

## Що я пропоную зробити

### 1. Визначити реальний ланцюг залежностей у `desktop/src-tauri`
Перевірити, який crate заводить `alloc-no-stdlib 3.0.0`, замість припущень.

### 2. Прибрати крихкий ручний пін з `Cargo.toml`
Якщо аналіз підтвердить, що прямий `alloc-no-stdlib` не потрібен як runtime dependency, замінити його на більш правильний підхід:
- або зафіксувати сумісний набір через `Cargo.lock`
- або використати точний dependency override/patch лише якщо це справді потрібно

### 3. Додати Rust lockfile в репозиторій для `desktop/src-tauri`
Для desktop-бінарника це найстабільніший шлях:
- локально й у CI використовуватиметься один і той самий набір Rust crate-ів
- збірка перестане “плавати” між різними версіями залежностей
- зникне проблема, коли GitHub Actions отримує інший graph, ніж локальна машина

### 4. За потреби скоригувати GitHub Actions
Якщо workflow зараз не гарантує відтворювану Rust-збірку, підправити його так, щоб він використовував lockfile і не підтягував інший dependency graph у CI.

### 5. Перевірити, чи немає другої помилки після brotli
Після усунення конфлікту dependency graph переглянути лог далі — часто перша помилка маскує наступну. Якщо є ще одна окрема причина падіння, виправити її в тій же ітерації.

## Технічні зміни
Ймовірно торкнуся таких файлів:
- `desktop/src-tauri/Cargo.toml`
- `.github/workflows/release.yml`
- можливо `desktop/src-tauri/Cargo.lock` (якщо його немає в репозиторії — додати)

## Очікуваний результат
Після цього збірка Windows у GitHub Actions має:
- перестати падати на `brotli` / `alloc-no-stdlib`
- стати відтворюваною між локальним середовищем і CI
- не залежати від випадкового оновлення crate-ів у майбутніх запусках
